# ad_deploy.ps1 — Active Directory Deployment Script

**Version:** 2.2 | **Minimum OS:** Windows Server 2012 R2

Automates Active Directory deployment for CDX-E exercises. Reads JSON configuration files from an exercise folder and idempotently deploys AD Sites, OUs, Security Groups, DNS zones, GPOs, Computer accounts, and User accounts against an existing domain — or creates a new AD forest from scratch on a bare server.

## Two Operating Modes

The script auto-detects which mode to enter based on whether an AD domain is reachable at runtime.

### Mode 1 — New Forest

When no domain is detected, the script:

1. Prompts for domain FQDN and NetBIOS name (or accepts them via parameters).
2. Installs the `AD-Domain-Services` Windows feature if not already present.
3. Creates a new AD forest with `Install-ADDSForest`.
4. Reboots the server (automatically with `-AutoReboot`, or manually by default).
5. On the next run (post-reboot), drops into Mode 2 to complete configuration.

> **Required privilege:** Local Administrator (pre-forest). Run as Domain Admin after promotion.

### Mode 2 — Post-Forest Configuration

When an existing domain is detected, the script proceeds directly to deploying AD objects from the exercise JSON files. Domain info is auto-detected from `Get-ADDomain` unless overridden via `-DomainFQDN` / `-DomainDN`.

> **Required privilege:** Domain Administrator.

## Prerequisites

### PowerShell Modules

The following modules must be available. They are installed automatically with the AD DS and DNS Server Windows features:

| Module | Feature | Required For |
|---|---|---|
| `ActiveDirectory` | RSAT-AD-PowerShell | Sites, OUs, Groups, Computers, Users |
| `ADDSDeployment` | AD-Domain-Services | Forest creation (Mode 1 only) |
| `DnsServer` | RSAT-DNS-Server | DNS zones and forwarders |
| `GroupPolicy` | GPMC | GPO creation and linking |

The script checks for these at startup and warns on any that are missing. Missing modules for phases that are not used (e.g., `DnsServer` if no DNS zones are defined) do not cause failures.

### Execution Policy

```powershell
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
```

Or run with `-ExecutionPolicy Bypass` if executing via a scheduled task or remote session.

## Required Configuration Files

All files must exist in the exercise folder (`EXERCISES\<ExerciseName>\`) before the script runs in Mode 2. The only exception is `structure.json`, which can be generated by `generate_structure.ps1`.

| File | Required | Purpose |
|---|---|---|
| `structure.json` | **Yes** | AD Sites, Subnets, Site Links, and OU hierarchy. Generated by `generate_structure.ps1` from `exercise_template.json`. |
| `users.json` | **Yes** | Security groups and user accounts. |
| `computers.json` | **Yes** | Computer account pre-staging with optional hardware metadata. |
| `gpo.json` | **Yes** | Group Policy Objects and OU link targets. |
| `services.json` | **Yes** | DNS primary zones and forwarders. |

If any file is missing, the script throws immediately before making any changes.

### structure.json — Key Schema

```json
{
  "sites": [
    { "name": "HQ-Site", "description": "Primary site" }
  ],
  "subnets": [
    { "cidr": "10.0.0.0/24", "site": "HQ-Site" }
  ],
  "siteLinks": [
    { "name": "HQ-BR-Link", "sites": ["HQ-Site", "Branch-Site"], "cost": 100, "replicationInterval": 180 }
  ],
  "ous": [
    { "name": "CDX", "dn": "OU=CDX", "description": "Root OU" },
    { "name": "Servers", "dn": "OU=Servers,OU=CDX", "description": "Member servers" }
  ]
}
```

OUs are sorted by depth automatically — parents are always created before children.

> The `Default-First-Site-Name` AD site is automatically renamed to the first entry in the `sites` array. `DEFAULTIPSITELINK` is removed.

### users.json — Key Schema

```json
{
  "groups": [
    {
      "name": "CDX-Admins",
      "scope": "Global",
      "category": "Security",
      "ou": "OU=Groups,OU=CDX",
      "description": "Exercise administrators"
    }
  ],
  "users": [
    {
      "samAccountName": "jsmith",
      "givenName": "John",
      "surname": "Smith",
      "ou": "OU=Users,OU=CDX",
      "title": "Systems Administrator",
      "department": "IT",
      "company": "ACME Corp",
      "mail": "jsmith@example.cdx.lab",
      "memberOf": ["CDX-Admins"]
    }
  ]
}
```

> `ou` values that do not include `DC=` components are automatically completed with the domain DN.

### computers.json — Key Schema

```json
{
  "computers": [
    {
      "name": "SRV-01",
      "ou": "OU=Servers,OU=CDX",
      "description": "Member server",
      "manufacturer": "Dell",
      "model": "PowerEdge R640",
      "service_tag": "ABCD1234"
    }
  ]
}
```

Hardware fields (`manufacturer`, `model`, `service_tag`) are optional. When present, they are stored as compact JSON in the AD `info` attribute. See [Hardware Metadata](#hardware-metadata-v21).

### gpo.json — Key Schema

```json
{
  "gpos": [
    { "name": "CDX-Baseline-Policy" },
    { "name": "CDX-Workstation-Policy" }
  ],
  "links": [
    {
      "gpoName": "CDX-Baseline-Policy",
      "targetOu": "OU=CDX",
      "enforced": false
    }
  ]
}
```

Both `gpoName`/`targetOu` and legacy `gpo`/`target` key names are supported for link definitions.

### services.json — Key Schema

```json
{
  "dns": {
    "zones": [
      { "name": "__AD_DOMAIN__", "type": "Primary" },
      { "name": "10.in-addr.arpa", "type": "Reverse" }
    ],
    "forwarders": [
      "192.168.1.1"
    ]
  }
}
```

The `__AD_DOMAIN__` placeholder is automatically replaced with the exercise domain FQDN at runtime.

## Parameters Reference

| Parameter | Type | Default | Description |
|---|---|---|---|
| `-ExerciseName` | String | *(prompted)* | Name of the exercise folder under `EXERCISES\`. Example: `CHILLED_ROCKET` |
| `-ExercisesRoot` | String | `.\EXERCISES` | Path to the folder containing all exercise subdirectories. |
| `-ConfigPath` | String | *(derived)* | Override the full path to the exercise folder. Skips `ExercisesRoot` + `ExerciseName` resolution. |
| `-DomainFQDN` | String | *(auto-detected)* | Domain FQDN override. Auto-detected from `Get-ADDomain` if omitted. Required during forest creation if not prompted interactively. |
| `-DomainDN` | String | *(derived)* | Domain Distinguished Name override. Derived from `-DomainFQDN` if omitted. |
| `-GenerateStructure` | Switch | `$false` | Regenerate `structure.json` from `exercise_template.json` before deployment. Calls `generate_structure.ps1` automatically. |
| `-WhatIf` | Switch | `$false` | Dry-run mode. Reports all planned changes without executing them. See [WhatIf](#whatif-dry-run). |
| `-AutoReboot` | Switch | `$false` | Enable automatic reboot after forest creation. See [AutoReboot Behavior](#autoreboot-behavior). |
| `-RebootDelaySeconds` | Int | `30` | Countdown duration (seconds) before the automatic reboot executes. Only meaningful with `-AutoReboot`. |

## Usage Examples

### Deploy an exercise against an existing domain

```powershell
.\ad_deploy.ps1 -ExerciseName "CHILLED_ROCKET"
```

### Regenerate structure.json and deploy in one step

```powershell
.\ad_deploy.ps1 -ExerciseName "CHILLED_ROCKET" -GenerateStructure
```

### New forest: create AD and auto-reboot to continue deployment

```powershell
.\ad_deploy.ps1 -ExerciseName "CHILLED_ROCKET" -GenerateStructure -AutoReboot
```

With a longer reboot countdown:

```powershell
.\ad_deploy.ps1 -ExerciseName "CHILLED_ROCKET" -GenerateStructure -AutoReboot -RebootDelaySeconds 60
```

### Preview all changes without making them

```powershell
.\ad_deploy.ps1 -ExerciseName "CHILLED_ROCKET" -WhatIf
```

### Override domain info (useful when auto-detection fails)

```powershell
.\ad_deploy.ps1 -ExerciseName "CHILLED_ROCKET" `
    -DomainFQDN "chilled-rocket.cdx.lab" `
    -DomainDN "DC=chilled-rocket,DC=cdx,DC=lab"
```

### Point to a non-standard exercises root

```powershell
.\ad_deploy.ps1 -ExercisesRoot "D:\CDX\EXERCISES" -ExerciseName "CHILLED_ROCKET"
```

### Supply the config path directly

```powershell
.\ad_deploy.ps1 -ConfigPath "C:\Exercises\CHILLED_ROCKET"
```

## Deployment Phases

After domain detection succeeds, the script executes the following phases in order. Each phase is idempotent — existing objects are skipped, not overwritten.

| Phase | Function | Source File | Objects Created |
|---|---|---|---|
| **1** | Sites, Subnets, Site Links | `structure.json` | AD Replication Sites; Replication Subnets; Site Links. Renames `Default-First-Site-Name`, removes `DEFAULTIPSITELINK`. |
| **2** | OU Hierarchy | `structure.json` | Organizational Units, sorted parent-first by DN depth. |
| **3** | Security Groups | `users.json` | Global/Universal/DomainLocal Security or Distribution groups. |
| **4** | DNS | `services.json` | Primary forward zones, reverse lookup zones, DNS forwarders. |
| **5** | Group Policy Objects | `gpo.json` | GPO stubs; GPO-to-OU links. Settings are NOT imported — GPO content must be configured separately. |
| **6** | Computer Accounts | `computers.json` | Pre-staged computer objects with optional hardware info in `info` attribute. |
| **7** | User Accounts | `users.json` | User accounts with all standard attributes; group memberships applied immediately after each user is created. |

## AutoReboot Behavior

When `-AutoReboot` is specified and a new forest is created, the script calls `Invoke-GracefulReboot`:

**Local console session:**

- Displays a countdown (default: 30 seconds) with a `Ctrl+C` cancel option.
- After reboot, the operator must manually re-run `ad_deploy.ps1` to continue.

**Remote PowerShell session (WinRM/PSRemoting):**

- Creates a Windows Scheduled Task named `CDX-PostReboot-Deployment` before rebooting.
- The task runs as `NT AUTHORITY\SYSTEM` at startup with a 2-hour execution time limit.
- The task re-launches `ad_deploy.ps1` automatically after the server comes back online.
- On the post-reboot run, the task is automatically detected and removed before deployment continues.

> If scheduled task creation fails (permissions issue, etc.), the script logs a warning and falls back to manual re-run instructions.

## Hardware Metadata (v2.1)

Computer accounts can carry physical hardware information stored as compact JSON in the AD `info` attribute. This enables asset tracking directly within AD without a separate CMDB.

**computers.json entry with hardware data:**

```json
{
  "name": "SRV-FINANCE-01",
  "ou": "OU=Servers,OU=CDX",
  "manufacturer": "Dell",
  "model": "PowerEdge R640",
  "service_tag": "7GHJK23"
}
```

**Resulting `info` attribute value:**

```json
{"manufacturer":"Dell","model":"PowerEdge R640","serviceTag":"7GHJK23"}
```

**Behavior:**

- New computers with hardware data: stored at creation time.
- Existing computers: `info` attribute is compared; updated only if the value has changed.
- Existing computers with no hardware data in `computers.json`: left untouched.

**Reading hardware info back:**

```powershell
$comp = Get-ADComputer "SRV-FINANCE-01" -Properties info
$comp.info | ConvertFrom-Json
```

## Idempotency

The script is safe to re-run at any time. For every object type, the script:

1. Attempts to retrieve the existing object.
2. If found — logs `(already exists)` in dark gray and skips.
3. If not found — creates the object and logs in green.

**Exception — Computer hardware info:** Existing computers are updated if their hardware data has changed. All other object attributes are not modified on re-run.

## WhatIf (Dry Run)

Adding `-WhatIf` performs a complete dry run without making any changes to Active Directory or DNS. All planned operations are printed to the console prefixed with `[WhatIf]`.

```powershell
.\ad_deploy.ps1 -ExerciseName "CHILLED_ROCKET" -WhatIf
```

`-WhatIf` is passed through to all AD and DNS cmdlets. Forest creation in Mode 1 is also skipped; the script logs what it would do and exits.

## Default User Password

All user accounts are created with the initial password:

```
P@ssw0rd123!
```

`ChangePasswordAtLogon` is set to `$true` on all accounts. Users must change their password on first login. Ensure your domain password policy permits this password complexity before running.

## Companion Script: generate_structure.ps1

`generate_structure.ps1` converts `exercise_template.json` into the `structure.json` file consumed by `ad_deploy.ps1`. It must be run (or invoked via `-GenerateStructure`) before the first deployment of any exercise.

```powershell
# Explicit run
.\generate_structure.ps1 -ExerciseName "CHILLED_ROCKET"

# Force regenerate even if structure.json already exists
.\generate_structure.ps1 -ExerciseName "CHILLED_ROCKET" -Force

# Run implicitly as part of ad_deploy.ps1
.\ad_deploy.ps1 -ExerciseName "CHILLED_ROCKET" -GenerateStructure
```

`generate_structure.ps1` must be located in the same directory as `ad_deploy.ps1`.
